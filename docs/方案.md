# 日志追踪 SDK 技术方案

## 1. 背景与问题

在微服务架构下，一次用户请求往往经过多个服务的协同处理。当出现故障或性能问题时，传统日志只记录单个服务内的信息，无法将跨进程的日志串联成一条完整的调用链，导致排查问题时需要在多个服务的日志中逐一比对时间戳——效率低下且容易遗漏。

核心挑战体现在三个方面：

- **链路断裂**：缺乏统一的请求标识符，同一请求在不同服务产生的日志无法关联
- **上下文丢失**：异步框架（如 asyncio）中，请求上下文难以在 `await` 调用链中自动传递
- **格式不统一**：各服务日志字段不一致，无法通过统一的查询条件定位全链路日志

本方案通过 `trace_id + span_id + parent_span_id` 三个字段来标记每一条日志的链路归属，借助 SDK 将这些字段自动注入到日志中，并通过 HTTP Header 在服务间传播，实现全链路日志的统一关联与定位。

---

## 2. 设计目标

| 目标 | 说明 |
|------|------|
| 零侵入接入 | 业务代码无需改动，注册中间件即可自动获得链路追踪能力 |
| W3C 标准兼容 | 遵循 W3C TraceContext 规范，traceparent Header 与主流 APM 工具互通 |
| 异步请求隔离 | 基于 asyncio ContextVar 实现，并发请求的上下文互不干扰 |
| 日志自动注入 | 每条日志自动携带 trace_id、span_id、parent_span_id，无需手动传参 |
| 跨服务传播 | aiohttp 出站请求自动注入传播头，下游服务自动继承同一 trace_id |
| 灵活导出 | 支持控制台、文件、OTLP 等多种 Span 导出方式，可按需切换 |

---

## 3. 核心方案：OpenTelemetry + Sanic 中间件 SDK

SDK 基于 OpenTelemetry 框架构建，以 Sanic 中间件为核心接入点，通过五个模块协同实现全链路日志追踪。

### 3.1 整体架构

| 模块 | 文件 | 职责 |
|------|------|------|
| 配置模块 | `config.py` | 定义 `TraceConfig` 数据类，统一管理服务名、导出方式、日志格式等所有参数 |
| 中间件模块 | `middleware.py` | `SanicTraceMiddleware` 核心类，在每个 HTTP 请求前后自动创建/结束 Span |
| 传播模块 | `propagation.py` | 从入站请求头提取父上下文，向出站请求头注入 traceparent |
| 日志集成模块 | `logging_integration.py` | `TraceContextFilter` 日志过滤器，自动向每条日志记录注入追踪字段 |
| 提供者与导出模块 | `provider.py` / `exporters.py` | 初始化全局单例 TracerProvider，管理多种 Span 导出器 |

整体工作流程如下：

1. 服务启动时，`SanicTraceMiddleware` 初始化全局 `TracerProvider`（单例），注册 W3C 传播器，并可选开启 aiohttp 自动插桩
2. 每个入站请求到达时，中间件从请求头中提取父上下文（如有），创建 SERVER 类型的 Span，并将其绑定到当前 asyncio 任务的 ContextVar 中
3. 请求处理期间，业务代码的所有 `logging` 调用经过 `TraceContextFilter` 时，自动从 ContextVar 读取当前 Span 信息并注入到日志字段中
4. 请求结束后，中间件结束 Span 并清理上下文；Span 数据经过处理器异步导出到配置的后端

### 3.2 Trace 字段模型

| 字段 | 位宽 | 编码格式 | 来源 | 根请求默认值 |
|------|------|---------|------|------------|
| `trace_id` | 128-bit | 32位小写十六进制字符串 | 链路入口处由 OTel SDK 随机生成；跨服务调用时从 traceparent Header 继承 | 随机生成 |
| `span_id` | 64-bit | 16位小写十六进制字符串 | 每个新 Span 创建时由 OTel SDK 随机生成 | 随机生成 |
| `parent_span_id` | 64-bit | 16位小写十六进制字符串 | 从当前 Span 的父 Span 中取得；若为根 Span 则无父 | 全零（`0000000000000000`） |

三个字段的关系：同一条请求链路中所有服务的日志共享同一个 `trace_id`；**每个服务（或业务层）拥有各自的 `span_id`；`parent_span_id` 记录上游 Span 的 ID**，从而形成可还原的树形调用关系。

字段命名与格式与 W3C TraceContext 标准完全对齐，可与 Jaeger、Tempo、Zipkin 等主流 APM 工具直接互通。

### 3.3 跨服务传播机制

**传播载体：traceparent Header**

跨进程传播使用 HTTP Header `traceparent`，格式为四段以短横线分隔的字符串，通过OpenTelemetry 提供的猴子补丁机制（AioHttpClientInstrumentor().instrument()）来自动注入transparent：

> 版本（固定为 `00`）- trace_id（32位 hex）- span_id（16位 hex）- 追踪标志（`01` 表示采样）

示例：`traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`

**入站处理（提取父上下文）**

当一个服务收到请求时，中间件的 `propagation` 模块会检查请求头中是否存在 `traceparent`：

- 若存在且格式合法：从中提取 `trace_id` 和上游的 `span_id`（作为本次 `parent_span_id`），并以此作为新 Span 的父上下文
- 若不存在或格式非法：视为链路入口，随机生成新的 `trace_id` 和 `span_id`

**响应头回写**

每个请求处理完成后，SDK 会将本次请求的 `trace_id` 写入响应头：

> `X-Trace-Id: {trace_id}`

通过 curl 或浏览器 Network 面板直接观察响应头，即可验证 traceparent 是否被正确提取和继承，无需查阅服务日志。

**出站处理（注入子上下文）**

SDK 提供两种出站注入方式：

- **aiohttp 自动插桩**（默认开启）：配置 `auto_instrument_aiohttp=True` 后，所有通过 aiohttp 发出的 HTTP 请求均由 OTel instrumentor 自动拦截并注入 `traceparent` 和 `baggage` Header，业务代码无感知
- **手动注入**：通过 `inject_trace_headers(headers)` 函数向任意请求的 headers 字典中注入当前上下文，适用于使用 requests、httpx 等其他客户端库的场景

**baggage 传播（预留）**

SDK 同时注册了 `W3CBaggagePropagator`，入站时会解析 `baggage` Header，出站时会自动注入。`TraceConfig.baggage_keys` 字段用于声明需要透传的业务字段（如 `user_id`、`tenant_id`），第一阶段默认为空列表，第二阶段填入实际字段后即可生效，无需改动传播层代码。

**多服务调用链示例**

以客户端 → 服务 A → 服务 B 的调用链为例：

1. 客户端发起请求，无 traceparent Header
2. 服务 A 中间件：生成 trace_id = `abc123`，span_id = `aaa111`，parent_span_id = 全零；日志携带这三个字段
3. 服务 A 业务代码通过 aiohttp 调用服务 B，SDK 自动在请求头中注入 `traceparent: 00-abc123-aaa111-01`
4. 服务 B 中间件：提取到 trace_id = `abc123`（继承），生成新 span_id = `bbb222`，parent_span_id = `aaa111`；日志同样携带这三个字段
5. 最终链路：所有日志的 trace_id 均为 `abc123`，通过 span_id 与 parent_span_id 可还原完整的调用树

### 3.4 日志自动注入

**工作原理**

`TraceContextFilter` 是 Python 标准 `logging.Filter` 的子类。SDK 初始化时将其挂载到日志处理器（handler）上。每次日志被记录时，过滤器会：

1. 调用 OTel API 获取当前 asyncio ContextVar 中绑定的活跃 Span
2. 从 Span 的上下文中读取 `trace_id` 和 `span_id`
3. 从 Span 的 `_parent` 属性中读取 `parent_span_id`
4. 将三个字段注入到当前日志记录（`LogRecord`）对象中

**请求隔离保证**

asyncio ContextVar 是 Python 3.7+ 的内置机制，其特点是：每个 asyncio Task 拥有独立的上下文副本，子任务继承父任务的上下文但不会反向影响父任务。这意味着即使服务同时处理数百个并发请求，每个请求的 trace_id 也完全隔离，不会出现串号问题。

**日志输出格式**

SDK 默认的日志格式为（可通过 `TraceConfig.log_format` 自定义）：

> `[时间戳] 日志级别 [trace_id=<值> span_id=<值> parent_span_id=<值>] [logger名称] 消息内容`

示例输出：

> `[2026-07-06 10:23:45,123] INFO [trace_id=4bf92f3577b34da6a3ce929d0e0e4736 span_id=00f067aa0ba902b7 parent_span_id=0000000000000000] [service-a.controller] 收到 GET /api/users 请求`

### 3.5 Span 导出方式

| 导出模式 | 配置值 | 适用场景 | 说明 |
|---------|-------|---------|------|
| 控制台输出 | `console` | 本地开发调试 | 将 Span 的 JSON 信息打印到标准输出，直观但量大 |
| 文件输出 | `file` | 离线分析、日志归档 | 每个 Span 以 JSON 行写入指定文件（JSONL 格式） |
| OTLP 上报 | `otlp` | 生产环境对接 APM | 通过 OTLP HTTP 协议将 Span 发送到收集器，默认端点为 `http://localhost:4318` |
| 不导出 | `none` | 仅需日志注入 | 只执行日志字段注入，不产生 Span 导出流量，适合轻量使用场景 |

OTLP 模式下，SDK 强制使用 `BatchSpanProcessor`（批量处理器），在后台异步聚合并发送 Span，最大限度降低对业务请求响应延迟的影响。

### 3.6 核心配置项

所有配置通过 `TraceConfig` 数据类传入，各字段说明如下：

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|-------|------|
| `service_name` | str | `"unknown-service"` | 服务名称，会作为 Span Resource 的 `service.name` 属性 |
| `exporter_type` | str | `"otlp"` | Span 导出方式，可选 `none` / `console` / `file` / `otlp` |
| `otlp_endpoint` | str | `"http://localhost:4318"` | OTLP HTTP 收集器地址，仅在 `exporter_type=otlp` 时生效 |
| `log_file_path` | str \| None | `None` | Span JSONL 文件导出路径，仅在 `exporter_type=file` 时生效 |
| `log_output_path` | str \| None | `None` | 日志文本落盘路径；设置后，带 trace_id 的日志同时写入文件，目录不存在时自动创建 |
| `log_max_bytes` | int | `10485760`（10 MB） | 单个日志文件的大小上限，超出后触发轮转 |
| `log_backup_count` | int | `5` | 日志轮转保留的历史文件数量 |
| `processor_type` | str | `"simple"` | Span 处理器类型：`simple`（同步，适合调试）/ `batch`（异步批量，适合生产） |
| `auto_instrument_aiohttp` | bool | `True` | 是否自动插桩 aiohttp 客户端，开启后出站请求自动注入 traceparent |
| `log_format` | str | 见 3.4 节 | Python logging 格式字符串，可自定义日志字段和顺序 |
| `log_level` | int | `10`（DEBUG） | 日志级别，遵循 Python logging 标准数值 |
| `resource_attributes` | dict | `{}` | 自定义 Resource 属性，合并到 Span 的元信息中（如 `env`、`version` 等） |
| `baggage_keys` | list[str] | `[]` | 需要通过 W3C baggage header 透传的业务字段名（如 `["user_id", "tenant_id"]`），第二阶段启用 |

## 4. 快速接入指南

### 4.1 最简接入

仅需两行初始化代码，无需改动任何业务逻辑：

```python
from sanic import Sanic
from log_middleware import SanicTraceMiddleware, setup_trace_logging
import logging

app = Sanic("my-service")
SanicTraceMiddleware(app, service_name="my-service")  # 注册中间件，自动开启 aiohttp 拦截
setup_trace_logging()                                  # 配置日志格式与过滤器

logger = logging.getLogger("my-service")

@app.get("/api/hello")
async def hello(request):
    logger.info("处理请求")  # 自动携带 trace_id、span_id、parent_span_id
    return sanic.response.json({"message": "hello"})
```

### 4.2 自定义配置

```python
from log_middleware import SanicTraceMiddleware, setup_trace_logging, TraceConfig

config = TraceConfig(
    service_name="order-service",
    exporter_type="otlp",
    otlp_endpoint="http://jaeger:4318",
    resource_attributes={"env": "production", "version": "1.2.0"},
    # 日志文本落盘，自动按大小轮转（目录不存在时自动创建）
    log_output_path="logs/order-service.log",
)

SanicTraceMiddleware(app, service_name="order-service", config=config)
setup_trace_logging(config=config)
```

### 4.3 分层架构（Controller → Service → Repository）

同一请求的所有 `await` 层级自动共享同一 trace_id，无需任何额外配置：

```python
from log_middleware import SanicTraceMiddleware, setup_trace_logging
import aiohttp, logging
from sanic import Sanic
import sanic.response

app = Sanic("service-a-layered")
SanicTraceMiddleware(app, service_name="service-a-layered")
setup_trace_logging()

repo_logger = logging.getLogger("service-a.repository")
svc_logger  = logging.getLogger("service-a.service")
ctrl_logger = logging.getLogger("service-a.controller")

class UserRepository:
    async def find_all(self):
        repo_logger.info("[repo] 查询本地用户数据")   # trace_id 与 controller 相同
        return [{"id": 1, "name": "Alice"}]

class UserService:
    def __init__(self):
        self._repo = UserRepository()

    async def get_users(self):
        svc_logger.info("[service] 开始聚合数据")     # trace_id 与 controller 相同
        local = await self._repo.find_all()
        # aiohttp 调用下游服务，SDK 自动注入 traceparent，无需手动处理
        async with aiohttp.ClientSession() as s:
            async with s.get("http://service-b/internal/data") as r:
                remote = await r.json()
        return {"local": local, "remote": remote}

_service = UserService()

@app.get("/api/users")
async def get_users(request):
    ctrl_logger.info("[controller] 收到请求")
    result = await _service.get_users()
    ctrl_logger.info("[controller] 返回响应")
    return sanic.response.json(result)
```

### 4.4 手动创建子 Span（增加链路粒度）

在关键业务逻辑处手动创建子 Span，可在 Jaeger 中看到更细粒度的耗时分布：

```python
from log_middleware import get_tracer

tracer = get_tracer(__name__)

class UserService:
    async def get_users(self):
        with tracer.start_as_current_span("UserService.get_users") as span:
            svc_logger.info("处理中")           # span_id 是子 Span 的 ID，trace_id 不变
            users = await self._repo.find_all()
            span.set_attribute("user_count", len(users))
            return users
```

## 5. 后端存储方案

SDK 通过 OTLP 协议将 Span 数据推送到后端收集器，支持对接以下后端：

### 5.1 OpenTelemetry + Jaeger

SDK 已内置 Jaeger 本地部署配置（`deploy/docker-compose.yml`），通过 Docker Compose 一键启动即可在本地完成全链路可视化验证：

- Jaeger UI 地址：`http://localhost:16686`
- OTLP HTTP 接收端口：`4318`（SDK 默认对接此端口）
- 存储引擎：Badger（本地持久化，适合开发/测试环境）

生产环境可将 Jaeger 替换为 Grafana Tempo，复用同一套 OTLP 导出配置，无需修改 SDK 接入代码。

### 5.2 后端存储选型

Span 和 Trace 数据量大、写多读少，适合使用列存或文档型数据库存储，以下为常见选型：

| 存储方案 | 优势 | 适用场景 |
|---------|------|---------|
| Elasticsearch | 全文搜索能力强，生态成熟 | 已有 ELK 基础设施的团队 |
| OpenSearch | 开源替代 ES，功能相近 | 希望规避 ES 授权问题的团队 |
| Cassandra | 写入吞吐量极高，天然分布式 | 超大规模数据量、高并发写入场景 |
| ClickHouse | 列存查询性能卓越，聚合分析快 | 需要对链路数据进行统计分析的场景 |

## 6. 公共 API 清单

| API | 类型 | 说明 |
|-----|------|------|
| `SanicTraceMiddleware` | 类 | 核心中间件，传入 Sanic app 和 service_name 即可启动追踪 |
| `TraceConfig` | 数据类 | 全局配置，所有参数均有合理默认值，按需覆盖 |
| `setup_trace_logging` | 函数 | 配置日志格式并挂载 TraceContextFilter |
| `get_tracer` | 函数 | 获取 Tracer 对象，用于手动创建子 Span |
| `inject_trace_headers` | 函数 | 向 headers 字典注入当前 traceparent，用于非 aiohttp 客户端 |
| `extract_trace_context` | 函数 | 从 Sanic 请求对象中提取 W3C 父上下文 |
| `TraceContextFilter` | 类 | Python logging 过滤器，可单独挂载到自定义 handler |
